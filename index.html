Claude 基礎 · 02
Anthropic 三大產品
底層模型相同，但入口、使用方式、能力範圍完全不同。
Ch
Claude（claude.ai）網頁 / App / 手機

通用 AI 對話助理。可問答、分析文件、寫作、生成 Artifacts、連接 MCP 工具。適合個人工作、學習、資料分析。

製作網站 / 工具：適合單一 HTML 檔案的網頁、快速原型、互動工具。Artifact 可即時在對話中渲染預覽，直接說「改這裡」就能修改，體驗流暢。複雜度高（多檔案、後端、資料庫）就不夠用了。
CC
Claude CodeCLI 終端機 + 桌機 App 整合

原本需要在終端機（Terminal）以 npm 安裝執行。直接讀取修改本機程式碼、執行 git、跑測試。深度整合開發工作流程，理解整個 Repo 脈絡。適合軟體開發者與工程師。

🆕 最新版桌機 App 已整合：Claude Desktop App 現在有 Chat / Code / Cowork 三個 Tab。Code tab 讓你透過 GUI 介面使用 Claude Code 全部功能，不需要開終端機。Code tab 目前仍在逐步推出，部分使用者可能尚未看到。
製作網站 / 工具：真正的開發首選。理解整個 Codebase、直接執行程式、安裝套件、跑測試除錯、操作 Git，整個開發循環一手包辦。多檔案、後端、框架（React/Vue）、資料庫都能處理。
CW
Claude Cowork桌機 App 整合 · Beta

桌機自動化工具。可操作本機視窗、整理檔案、執行文件任務。「你說目標，它去電腦上完成」，每步驟截圖識別再決定下一步。需要 VM 沙箱環境（Windows: VM Platform；Mac: Apple Silicon M1+）。

🆕 最新版桌機 App 已整合：Cowork 也整合進 Claude Desktop App，在同一個 App 內切換 Tab 即可使用，不需要另外下載獨立工具。底層與 Code 使用相同引擎（Claude Code 架構），但介面設計給非技術知識工作者使用。
製作網站 / 工具：可以生成 .html 存到本機，但沒有即時預覽，也不是設計目標。Cowork 的強項是辦公室自動化任務，而不是開發工作。
🆕 Claude Desktop App：三合一介面（2026 最新）
最新版 Claude Desktop App 已將三個產品整合為同一個 App 內的三個 Tab：Chat ／ Code（</>） ／ Cowork，可直接在 App 內切換，不需要分別安裝不同工具。
⚠ 網頁版 claude.ai 只有 Chat，無法切換到 Code 或 Cowork Tab，這兩個功能僅限桌機 App 使用。

Windows VM Platform 是什麼？為什麼需要？
VM Platform 是 Windows 內建的虛擬化功能，也是 WSL2、Windows Sandbox 的底層技術。Cowork 需要它在你的電腦上啟動一個輕量 Linux 虛擬機，作為 Claude 跑程式碼和處理檔案的隔離環境。即使 Claude 執行出錯，也不會影響你真實的作業系統。

Claude 基礎 · 04
Skill 是什麼？
「Skill」在 Claude 生態裡出現在三個不同地方，指的是完全不同的東西。
① claude.ai Agent Skills — 付費使用者可用
claude.ai 的 Skill 分兩種：預建 Skills（Anthropic 官方提供，如 skill-creator、PPT、Word、PDF skill，付費用戶開箱即用）和自訂 Skills（使用者自己上傳 ZIP 檔，透過 Settings → Features，Pro 以上方案可用）。
✨ skill-creator（預建）

把你剛才在對話中完成的任務包成可重複使用的 Skill。你不需要是開發者，對話完成後直接說「把這個包成 Skill」即可。

📦 自訂 Skill（上傳）

將自己寫好的 SKILL.md 打包成 ZIP，透過 Settings → Features 上傳。之後 Claude 在相關任務時會自動載入或透過斜線指令呼叫。

② Cowork Skill — 本機 SKILL.md 檔案
存放在本機 /mnt/skills/ 資料夾（或 Copilot Cowork 的 OneDrive）的 Markdown 檔案。定義「如何完成一項任務」的步驟、工具調用方式、輸出格式。Cowork 在每次對話開始時自動載入。使用者可以自己編寫和修改，也可以用 skill-creator 從對話中建立。
③ API 層 Skill — 開發者 System Prompt 設定
開發者在 API 程式碼層設定 System Prompt，預先定義 Claude 的角色、工具、輸出格式。使用者完全看不到這層設定。適合建立企業內部的專用 AI 應用。

功能模組 · 05
Chat · Projects · Artifacts · Customize
claude.ai 的四個核心功能模組，各有不同定位與使用情境。
💬 Chat（對話）最核心

最基礎的對話介面。每個 Chat 是獨立情境，不跨 session 保留記憶（除非開啟記憶功能或使用 Projects）。

🔎 Research
深度研究模式。自動執行多輪搜尋、交叉比對多個來源，花數分鐘產出有引用來源的完整報告。適合「深入研究某主題」，不適合簡單問答。
📎 Add files or photos
手動上傳本機檔案給 Claude 閱讀分析。這是「你選檔案給 Claude 看」，不是「Claude 去你電腦裡操作檔案」。後者只有 Cowork 做得到。
📁 Projects（專案）Pro / Team / Enterprise

整合多個 Chat 至同一專案，可上傳背景文件（PDF/Word 等）、設定自訂指令。Claude 在此專案所有對話都能記住設定與文件。適合法律審查、品牌指南等長期重複工作情境。

共享：Free / Pro 個人專屬 ／ Team 可建立團隊共用 Project ／ Enterprise 加上完整管理控制
🖼 Artifacts（成品）

Claude 在對話中生成並在右側面板即時渲染的獨立內容。支援格式：HTML、React、SVG、Markdown、程式碼。PPT/Word 等二進位檔不支援。

Artifacts 分頁裡看到的是什麼？
左側 Artifacts 分頁顯示的是你已 Publish 的 Artifacts，不是所有產出過的 Artifact。未 Publish 的 Artifact 存在原始對話的右側面板，要找回就回到那段對話。對話刪除後即消失。
🎨 Customize（個人化）

在 Settings 中設定回應風格、語言偏好、格式習慣（永遠用繁中、回答簡短等），也包含記憶管理（Memory）功能。

Artifact 分頁與 Publish 說明
📂 Artifacts 分頁

左側 Artifacts 分頁顯示的是你已 Publish 的 Artifacts，不是所有產出過的 Artifact。未 Publish 的 Artifact 只存在於原始對話的右側面板，回到那段對話才看得到。對話刪除後即消失。

🌐 Publish 功能

把 Artifact 發佈成公開網址讓別人瀏覽。目前只有 Public，沒有私人保存選項。想私人保存：複製程式碼存到本機，或使用 create_file 產出的 HTML 檔案下載保存。

Artifact 產出 → 存在對話右側面板（未 Publish）→ 點 Publish 後才出現在左側 Artifacts 分頁。show_widget 和 create_file 的產出無論是否 Publish 都不會進入 Artifacts 分頁。
Copilot 生態 · 09
Microsoft Copilot 產品家族
M365 Copilot · Copilot Cowork · Copilot Studio — 三個不同定位的產品。
💬 M365 Copilot$30/人/月

嵌入 Word/Excel/PPT/Outlook/Teams 的 AI 側欄助理。一次只能在單一 App 內工作。個別員工日常生產力提升用。

🚀 Copilot CoworkWave 3，含在 M365 Copilot 內Frontier Beta

🔤 Token 與效能經濟效益
了解 Token 的本質，用對方法讓 Claude 更聰明、更省、更快。
🔤 Token 是什麼？

Token 是 AI 模型處理文字的最小單位，不等於字數。中文約每 1 個字 = 1–2 tokens；英文約每 4 個字母 = 1 token。Claude 處理你輸入的訊息（input tokens）再產出回答（output tokens），兩者合計就是這次對話消耗的 Token 量。

📥
Input Token
你傳給 Claude 的一切：
問題 + 上傳文件 + 對話歷史 + System Prompt
📤
Output Token
Claude 回覆的內容。
通常比 input 少，但長篇回答也會很大
🧠
Context Window
Claude 同時能記住的上限。
超過就「忘記」最早的內容
⚡ Token 影響什麼？
面向說明實際感受
訊息用量上限 claude.ai 按 每條訊息消耗的 Token 量 計算用量，不是純看「幾條訊息」 傳一個 100 頁 PDF 再問問題，比問一句話消耗多很多
回答品質 對話越長，Claude 要同時處理的歷史 Token 越多，有時推理品質會下降 長對話後期 Claude 可能「繞圈圈」或忘記早期細節
回應速度 Input Token 越多，處理時間越長 貼大量程式碼或長文件，第一個字回來會比較慢
API 費用 直接按 input + output token 計費 開發者自建應用時要特別注意 System Prompt 大小
💡 效能 + 經濟效益：核心技巧
✂️ 技巧一：同個 Project，開新 Session（最重要）推薦做法

對話越長，每一輪都要把整段歷史重新塞進 Context Window。這會讓 Token 消耗快速累積，也可能讓 Claude 被雜訊干擾、品質下降。

❌ 低效做法
同一個 Session 裡問了 50 輪對話
→ 第 50 輪時 Claude 要處理前 49 輪的全部歷史
→ Token 暴增、速度慢、回答品質可能下降
✅ 推薦做法
同個 Project 下開多個 Session
→ Project 的背景文件與指令持續生效
→ 每個 Session 從乾淨狀態出發，更聚焦
具體做法：把長期背景（角色定義、公司資訊、寫作風格）放進 Project 的自訂指令或上傳文件。每次新任務就在同個 Project 開新 Chat，不用重新貼背景，也不帶著上次的雜訊。
🎯 技巧二：問題說清楚，一次到位

模糊的問題 → Claude 猜測 → 你修正 → 再問一輪，來回幾次累計的 Token 遠比一開始說清楚多。描述任務時加上：背景、目的、格式要求、字數，可以大幅減少來回次數。

📄 技巧三：只上傳相關部分

貼整份 100 頁報告只為了問第 3 頁的數字，等於讓 Claude 讀了 97 頁的雜訊。盡量只貼相關段落，或先讓 Claude 知道要看哪裡，再選擇性貼上。

🔄 技巧四：長對話定期「重置總結」

如果某個 Session 不得不聊很長，可以請 Claude 在中途「整理一份結論摘要」，然後把摘要貼到新 Session 繼續，既省 Token 又讓 Claude 重新聚焦。

📝 技巧五：善用 Project 自訂指令

把每次都要重複說的背景設定（「你是我們公司的法律顧問，請用繁中回答，格式為條列式…」）寫進 Project 的自訂指令，後續每個 Session 自動套用，不用每次都貼。

📋 Token 效能速查
情境問題根源建議做法
Claude 越問越「跑題」對話歷史太長，早期指令被稀釋在同 Project 開新 Session
訊息額度用得很快每輪都帶著龐大的歷史 Token縮短單次 Session 長度，只貼相關內容
每次都要重新說背景沒有使用 Project 功能建立 Project，設定自訂指令 + 上傳背景文件
回答速度很慢Input Token 過多（大文件 + 長歷史）只貼相關段落，必要時開新 Session
Claude 忘記前面說的事超過 Context Window 上限整理摘要後開新 Session 繼續
最佳實踐總結：
Project 是你的「長期記憶庫」（背景、指令、文件）
Session 是每次「工作任務的執行單位」（聚焦、乾淨、不帶雜訊）
→ 用 一個 Project + 多個短 Sessions，是最省 Token、品質最穩的使用方式。
